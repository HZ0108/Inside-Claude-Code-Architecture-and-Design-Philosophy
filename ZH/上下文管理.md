## 1. 概述

Claude Code 的上下文管理系统是该工程最核心、最复杂的子系统之一。它由多个层次的机制组成，从底层的 Token 计数，到中层的微型压缩（工具结果裁剪），再到高层的全量压缩（对话历史摘要），最后到会话级和持久化级的记忆管理。整个系统围绕一个核心目标：**在有限的上下文窗口内最大化有效信息量**。

### 1.1 架构层次总览

```
┌──────────────────────────────────────────────────────┐

│  Layer 5: 持久化记忆提取 (extractMemories)            │

│            → 跨会话持久化到 ~/.claude/projects/       │

├──────────────────────────────────────────────────────┤

│  Layer 4: 会话记忆 (Session Memory)                  │

│            → 当前会话的 Markdown 笔记                 │

├──────────────────────────────────────────────────────┤

│  Layer 3: 全量压缩 (Compaction)                      │

│            → AI 摘要历史对话（调用模型）               │

├──────────────────────────────────────────────────────┤

│  Layer 2: 历史裁剪 (Snip)                            │

│            → 选择性丢弃历史（按时间/Token）            │

├──────────────────────────────────────────────────────┤

│  Layer 1: 微型压缩 (Microcompact)                    │

│            → 清理旧工具结果，保留提示缓存              │

├──────────────────────────────────────────────────────┤

│  Layer 0: Token 计数与阈值检测                        │

│            → 一切决策的数学基础                        │

└──────────────────────────────────────────────────────┘
```

### 1.2 系统设计原则

- **分层管理**：每层机制处理不同粒度的问题（Token级 → 工具级 → 消息级 → 会话级）
- **保守触发**：所有自动机制都配有熔断器，防止反复失败浪费 API 调用
- **零信息丢失意图**：通过结构化摘要而非简单丢弃来压缩历史
- **提示缓存友好**：压缩操作优先复用已有的 API 提示缓存
- **渐进式优化**：优先使用轻量级机制（MC → Snip → 压缩），重型机制兜底

## 2. Token 计数体系

Token 计数是整个上下文管理的基础，所有压缩决策都基于 Token 数量。

### 2.1 精确计数 vs. 估算

Claude Code 使用两种 Token 计数策略，各有适用场景：

#### 精确计数（API 调用后）

```typescript
// utils/tokens.ts

getTokenUsage(message: Message): Usage | undefined

// 从 assistant 消息的 usage 字段中提取

// 条件：消息有 usage 字段 + 不是合成消息 + 不是 SYNTHETIC_MODEL

// 不返回 progress/attachment/system 类型的消息
```

`Usage` 对象包含：
```typescript
{

  input_tokens: number,

  output_tokens: number,

  cache_creation_input_tokens?: number,  // 创建缓存消耗

  cache_read_input_tokens?: number,        // 读取缓存节省

}
```

#### 精确总上下文窗口 Token 数

```typescript
// utils/tokens.ts

getTokenCountFromUsage(usage: Usage): number

// = input_tokens + cache_creation_input_tokens + cache_read_input_tokens + output_tokens

// 这个数字代表这次 API 调用时的完整上下文大小
```

#### 从历史消息中获取精确计数

```typescript
// utils/tokens.ts

tokenCountFromLastAPIResponse(messages: Message[]): number

// 向前遍历消息数组，找到第一个有 usage 的 assistant 消息

// 返回该消息的 getTokenCountFromUsage

// 用于压缩前后的 token 数量对比
```

#### 用于 task_budget 的最终窗口 Token

```typescript
// utils/tokens.ts

finalContextTokensFromLastResponse(messages: Message[]): number

// 从 usage.iterations[-1] 获取最终窗口大小

// （用于 server-side tool loop 场景）

// 公式：input_tokens + output_tokens（不含 cache tokens）
```

### 2.2 粗略估算（无精确数据时）

```typescript
// services/tokenEstimation.ts

roughTokenCountEstimation(content: string, bytesPerToken = 4): number

// 最简单的估算：字符数 ÷ 4

// 适用于纯文本内容
```

**按文件类型调整比率**：
```typescript
// JSON/JSONL/JSONC 文件的字节/Token 比接近 2（因为大量单字符符号）

// 其他文件使用默认的 4

bytesPerTokenForFileType(fileExtension: string): number {

  switch (fileExtension) {

    case 'json': case 'jsonl': case 'jsonc': return 2

    default: return 4

  }

}
```

### 2.3 混合计数（标准函数）

```typescript
// utils/tokens.ts

tokenCountWithEstimation(messages: readonly Message[]): number
```

这是**最重要的计数函数**，是自动压缩阈值的判断依据。逻辑如下：

==**PS**：能精确计算的就精确计算，不能精确计算的部分再进行单独粗略计算==
```
1. 从最后一条消息向前遍历
2. 找到第一个有精确 usage 数据的 assistant 消息
3. 若找到：
   a. 先向后回溯，将所有相同 message.id 的 split 消息纳入估算

      （并行 tool calls 时，同一 API 响应产生多条 assistant 消息）

   b. 精确 usage + 之后消息的粗略估算
4. 若未找到：全部使用粗略估算
```

**并行 tool calls 的处理**：当模型在一个响应中并行调用多个工具时，streaming 代码为每个 content block 发射单独的 assistant 消息记录（共享 `message.id` 和 `usage`），并在各 tool_use 之间立即插入 tool_result。消息数组结构如下：

```
[..., assistant(id=A, usage), user(tool_result_A), assistant(id=A, usage),

 user(tool_result_B), assistant(id=A, usage), user(tool_result_C), ...]
```

若只在最后一条 assistant 记录停止，会漏掉所有中间插入的 tool_result，导致低估。`tokenCountWithEstimation` 的回溯逻辑将所有共享 `message.id` 的记录都纳入估算。

PS：
```
- **常规计算逻辑的盲区：** 很多统计系统在计算一轮对话的 Token 消耗或组装历史消息时，习惯性地**“寻找最后一条 assistant 消息”**作为当前回合的结束点。
    
- **低估发生的原因：** 如果系统只抓取最后那条 `assistant(id=A, usage)` 来结算，它就会无视掉前面那些同属于 `id=A` 的 `assistant` 记录，更会漏掉夹在中间的巨大文本——也就是那些 `tool_result`（工具返回的结果通常包含大量 Token）。这就导致了系统算出来的 Token 数量比实际消耗的要少得多（低估）。
```

### 2.4 特殊内容类型的估算

```typescript
// services/tokenEstimation.ts

roughTokenCountEstimationForBlock(block): number

// 文本块：字符数 ÷ 4

if (block.type === 'text')

  return roughTokenCountEstimation(block.text)

// 图像/文档块：固定 2000 Token

// 原因：Claude 平台对图片的计费公式为 (width × height) / 750

// 上限 2000×2000 = 5333 tokens，取保守值 2000

if (block.type === 'image' || block.type === 'document')

  return 2000  // IMAGE_MAX_TOKEN_SIZE

// tool_use：工具名 + JSON化的输入参数

// 注意：这里的 input 可能是任意大（bash 命令、文件内容等）

// 直接 stringify 一次来估算，因为 API 也会重新序列化

if (block.type === 'tool_use')

  return roughTokenCountEstimation(

    block.name + JSON.stringify(block.input ?? {})

  )

// tool_result：递归计算其内容

if (block.type === 'tool_result')

  return roughTokenCountEstimationForContent(block.content)

// thinking 块：只计算 thinking 文本，不含 JSON 包装和签名

if (block.type === 'thinking')

  return roughTokenCountEstimation(block.thinking)

// 已编辑的 thinking：计算 data 字段

if (block.type === 'redacted_thinking')

  return roughTokenCountEstimation(block.data)

// 其他所有块类型（server_tool_use、web_search_tool_result 等）

// 统一 stringify 后估算

return roughTokenCountEstimation(JSON.stringify(block))
```

## 4. 微型压缩（Microcompact）

微型压缩是每次 API 调用前执行的最轻量级上下文管理机制。它==不进行摘要，而是直接清理或替换旧工具结果的内容==。

==**PS：** 其实是做剪枝==

### 4.1 两种触发模式

#### 4.1.1 时间触发（Time-Based Microcompact）

**核心思想**：Claude API 的提示缓存（Prompt Cache）有 1 小时（60 分钟）的 TTL。当距离上次 Assistant 消息超过 60 分钟时，**缓存已过期**，整个前缀会被重新计算。此时，**提前清理旧的工具结果，可以减少重写的数据量**。

==PS：缓存是否命中除了看前缀是否相同，一定要看看TTL的情况。但是这个TTL是统计两次请求之间的时间差来计时==

```typescript
// services/compact/timeBasedMCConfig.ts

const TIME_BASED_MC_CONFIG_DEFAULTS = {

  enabled: false,               // 默认关闭，通过 GrowthBook 'tengu_slate_heron' 启用

  gapThresholdMinutes: 60,    // 60 分钟（与服务器缓存 TTL 对齐）

  keepRecent: 5,               // 保留最近 5 个可压缩工具结果

}
```

**触发判断**：
```typescript
evaluateTimeBasedTrigger(messages, querySource): { gapMinutes, config } | null

// 必须满足：

// 1. config.enabled = true（GrowthBook 开关）

// 2. querySource === 'repl_main_thread'（仅限主线程）

// 3. 有最近一条 assistant 消息

// 4. gap = (now - lastAssistant.timestamp) / 60000 >= config.gapThresholdMinutes
```

**处理逻辑**（`maybeTimeBasedMicrocompact`）：
1. `collectCompactableToolIds()` 收集所有**可压缩工具**的 tool_use_id（按出现顺序） -> ==也是设置了哪些工具的结果是可压缩的==
2. 保留最近 N 个（`keepRecent`），其余加入 `clearSet` -> 剪枝保护
3. 遍历消息，将 `clearSet` 中工具的 tool_result 内容替换为 `[Old tool result content cleared]`
4. **关键**：修改提示内容会破坏服务器端缓存，因此重置 Cached MC 状态并通知缓存断裂检测

==PS：这个也是直接一把进行剪枝，比openclaw用窗口守卫实现动态平衡的方法更合理==

**清理范围**（`COMPACTABLE_TOOLS`）：
```typescript

  'Grep',

```
==AgentTool、TaskOutputTool 等关键工具**不在此列表**，不会被清理==。

#### 4.1.2 缓存编辑（Cached Microcompact）

**核心思想**：利用 API 的 `cache_edits` 机制，通过在请求中指定 `cache_reference`（引用已有缓存片段）和 `cache_edits`（编辑操作），删除旧工具结果，同时**保留提示缓存的有效性**。

```
┌──────────────────────────────────────────────────────────────┐

│  服务器端缓存（Prompt Cache）                                 │

│                                                              │

│  [系统提示 + 工具定义 + 消息前缀]  ← 已缓存，cache_hit        │

│                                                              │

│  [工具结果A: Read foo.py]           ← cache_edit: 删除        │

│  [工具结果B: Bash git status]       ← cache_edit: 删除        │

│  [工具结果C: Grep "func"]           ← 保留                    │

│  [最新用户消息]                                            │

└──────────────────────────────────────────────────────────────┘

                              ↓

                    API 请求只传输 delta

                    （删除哪些，而不是全部）
```

PS：

==实际优势：cache_edit进行删除，相当于mask住了部分KV，但是KV缓存没被破坏==

这些内容是在session建模中考虑的字段，context中没有这些内容（所有infra层指令参数这些都在session中，不在context中）

**启用条件**（`microCompact.ts` 中的 `cachedMicrocompactPath`）：
1. Feature flag `CACHED_MICROCOMPACT` 开启
2. 模型支持缓存编辑（`isModelSupportedForCacheEditing(model)`） -> ==检查模型服务是否支持该功能(由infra侧决定)==
3. `isMainThreadSource(querySource)` 为 true（`querySource === 'repl_main_thread'` 或以此前缀开头）

**实现状态追踪**（`cachedMicrocompact.js`，动态导入）：
```typescript
CachedMCState = {

  toolOrder: string[],           // 所有注册过的 tool_use_id（按顺序）

  registeredTools: Set<string>,  // 已注册的 tool_use_id

  deletedRefs: Set<string>,     // 已删除的 tool_use_id

  pinnedEdits: PinnedCacheEdits[],  // 钉住的编辑操作

}
```

**工作流程**：
1. **注册**：每条 user 消息中，对每个新的 tool_result，调用 `registerToolResult(tool_use_id)`，并按 user message 分组
2. **判断**：调用 `getToolResultsToDelete()`，基于 GrowthBook 配置的阈值判断哪些该删除
3. **钉住**：若检测到有编辑（编辑类工具的 tool_result），将编辑操作钉住（`pinCacheEdits`），确保下次请求也发送
4. **执行**：创建 `cache_edits` block，标记要删除的 tool_use_id
5. **API 层注入**：在 `microCompactMessages` 返回后，API 层添加 `cache_reference` 和 `cache_edits`

### 4.2 微压缩决策流程

```typescript
// services/compact/microCompact.ts

microcompactMessages(messages, toolUseContext?, querySource?): MicrocompactResult

// 执行顺序（短路逻辑）：

async function microcompactMessages(messages, toolUseContext, querySource) {

  clearCompactWarningSuppression()  // 清除警告抑制

  // 1. 时间触发 MC 优先（检测到缓存已过期）

  const timeResult = maybeTimeBasedMicrocompact(messages, querySource)

  if (timeResult) return timeResult  // 短路：时间触发优先

  // 2. 缓存编辑 MC（需满足条件）

  if (feature('CACHED_MICROCOMPACT') && isCachedMCEnabled() && isModelSupported() && isMainThread) {

    return cachedMicrocompactPath(messages, querySource)  // 短路

  }

  // 3. 无可用 MC 路径 → 返回原消息（fallback）

  return { messages }

}
```

## 5. Snip 历史裁剪（这个功能类似于截断，==默认是没开启的==）

Snip（`HISTORY_SNIP` feature flag）是一种在消息历史层面直接丢弃早期内容而不进行摘要的机制。与压缩不同，Snip 丢弃的是模型不需要再看到的内容（如已经被处理完的文件读取结果）。

### 5.1 工作原理

```
消息历史

  [消息0] [消息1] ... [消息N-2] [消息N-1] [最新消息]

      ↑                              ↑

   snip点 ← ← ← ← ← ← ← ← ← ← ← ← ← 保留尾部

Snip 后：

  [SNIP_MARKER] [消息N-2] [消息N-1] [最新消息]
```

### 5.2 与压缩的关系

- ==**可以共存**：Snip 和压缩并不互斥，可以同时触发==
- ==**执行顺序**：Snip 在微压缩**之前**执行（`query.ts:396`）==
- **Token 释放追踪**：`snipTokensFreed` 被传递给 `autoCompactIfNeeded`，确保自动压缩的阈值判断反映 Snip 已释放的空间

```typescript
// query.ts 中：

if (feature('HISTORY_SNIP')) {

  const snipResult = snipModule!.snipCompactIfNeeded(messagesForQuery)

  messagesForQuery = snipResult.messages

  snipTokensFreed = snipResult.tokensFreed

  if (snipResult.boundaryMessage) {

    yield snipResult.boundaryMessage

  }

}

// ... 后续 microcompact 和 autocompact 使用 snipTokensFreed
```

### 5.3 Snip 的保留策略

Snip 不会无差别丢弃，而是保留：
1. **最近 N 条消息**：确保工作上下文不被破坏
2. **包含 tool_use 尚未配对 tool_result 的消息**：防止孤儿 tool_result
3. **会话边界消息**：如压缩边界、SessionStart 消息等

PS：

直接修改 messages[]，把中间一段替换成一个占位标记，然后只发后面的部分给 API -> 相当于把头部截断了

 是编译后的动态模块，通过 require() 延迟加载，所以不知道具体的推测触发时机，从代码结构来看应该是按量触发的（另外可以用`/force-snip`命令由用户手动触发）

### 7.3 压缩摘要提示词设计

压缩摘要要求模型生成极度结构化的输出，这是整个压缩系统最关键的部分之一。

#### 7.3.1 提示词模板结构

提示词包含三个部分：

**1. 强制性无工具前导（CRITICAL）**

```
CRITICAL: Respond with TEXT ONLY. Do NOT call any tools.
- Do NOT use Read, Bash, Grep, Glob, Edit, Write, or ANY other tool.
- You already have all the context you need in the conversation above.
- Tool calls will be REJECTED and will waste your only turn — you will fail the task.
- Your entire response must be plain text: an <analysis> block followed by a <summary> block.
```


```
**极度重要：**仅回复纯文本。切勿调用任何工具。
- 切勿使用 Read、Bash、Grep、Glob、Edit、Write 或任何其他工具。
- 您已经在上述对话中获得了所需的所有上下文。
- 工具调用将被拒绝，并且会浪费您仅有的一次轮次——这将导致您的任务失败。
- 您的完整回复必须是纯文本：一个区块，后跟另一个区块。
```

**设计理由**：压缩的 fork 子 Agent 继承了父会话的完整工具集，==但模型可能仍然尝试工具调用。这段前导明确禁止工具使用，确保模型只输出文本==。

PS：之前测openclaw的时候是有发现模型有概率不follow压缩指令，而选择继续执行后续任务


**2. 详细分析指令（DETAILED_ANALYSIS_INSTRUCTION）**

要求模型在 `<analysis>` 块中做以下事情：
- 按时间顺序分析每条消息 -> 克服模型在长对话中的“注意力丢失”和“上下文遗忘”问题，防止逻辑断层
- 识别：显式请求、意图、代码片段、文件名、函数签名 -> 实体提取，转成结构化信息
- 记录：遇到的错误、用户纠正、不再尝试的方法  -> 负向记忆，防止反复踩坑
- 确保技术准确性和完整性

**3. 摘要格式要求（9 个必需部分）**

PS：openclaw这种更通用的agent，其他几个部分都类似，但**重要代码片段**等换成了**重要上下文片段**

```
1. Primary Request and Intent:
   用户的全部显式请求和意图

2. Key Technical Concepts:
   重要技术概念、框架、模式列表

3. Files and Code Sections:
   - 文件名
     - 该文件为何重要
     - 对该文件做了什么修改（如果有）
     - 重要代码片段
   - ...

4. Errors and Fixes:
   - 错误描述：
     - 修复方法
     - 用户对该错误的反馈（如果有）
   - ...

5. Problem Solving:
   已解决的问题 + 正在进行的排查

6. All User Messages:
   所有非工具结果的用户消息（按时间顺序）

7. Pending Tasks:
   显式要求但尚未完成的任务

8. Current Work:
   压缩前正在进行的具体工作

   （包含文件名、代码片段）

9. Optional Next Step:
   直接承接最近工作的下一步
```

## 8. 会话记忆（Session Memory）

会话记忆是一种**预防性**上下文管理机制。它在对话进行过程中，在后台持续维护一份 Markdown 笔记，**将关键信息提取出来**。这使得压缩操作可以将整个对话历史替换为「会话笔记 + 保留的尾部消息」，同时保留大量上下文。 -> 上下文信息卸载到文件系统

### 8.1 与压缩的核心区别

|        | 全量压缩     | 会话记忆               |

| ------ | -------- | ------------------ |

| 时机     | 上下文压力过大时 | 对话进行中持续维护          |

| 方式     | 从零生成摘要   | **增量更新**已有笔记       |

| 内容     | 每次压缩重新生成 | 逐步积累               |

| API 调用 | 需要       | 需要（**子 Agent 更新**） |

| 信息覆盖   | 依赖模型摘要质量 | 增量积累，更全面           |

### 8.2 触发条件

```typescript
// sessionMemoryUtils.ts

DEFAULT_SESSION_MEMORY_CONFIG = {

  minimumMessageTokensToInit: 10_000,    // 初始化：上下文至少 10K token

  minimumTokensBetweenUpdate: 5_000,     // 更新：距上次至少增长 5K token

  toolCallsBetweenUpdates: 3,            // 更新：至少 3 次工具调用

}
```

**触发条件**（需**同时满足**）：
```typescript
shouldExtractMemory(messages: Message[]): boolean {

  // 1. 已初始化（已满足 minimumMessageTokensToInit）

  // 2. Token 增长 >= minimumTokensBetweenUpdate

  // 3. 工具调用次数 >= toolCallsBetweenUpdates

  // 4. 最后一次 Assistant 消息无工具调用（自然对话停顿点）

  const shouldExtract =

    (hasMetTokenThreshold && hasMetToolCallThreshold) ||

    (hasMetTokenThreshold && !hasToolCallsInLastTurn)

}
```

**为何要求"最后无工具调用"**：这确保提取发生在自然对话间隙，而非工具调用中间，避免遗漏上下文。

PS：
**场景一：Sub-agent 更新会话记忆时**

## ==9. 会话记忆压缩（Session Memory Compaction，重要）==

会话记忆压缩（SM-Compact）是全量压缩的**替代方案**，用增量积累的会话笔记替代每次从零生成的摘要。

### 9.1 与传统压缩的对比

| 维度 | 传统压缩 | SM-压缩 |

|-----|--------|--------|

| 摘要来源 | 每次从零生成 | 增量维护的会话笔记 |

| 历史处理 | 全部替换为摘要 | 选择性保留尾部 |

| API 调用 | 需要（调用模型生成摘要） | **不需要** |

| 保留内容 | 仅摘要文本 | 摘要 + 最近 10K-40K Token |

| 适用场景 | 通用 | 长时间多轮次会话 |

PS：
**==Session Memory Compact 比 传统压缩的优先级更高==**，Session Memory Compact 失败时回退到传统压缩

Session Memory Compact 的优势：
```
  传统 compactConversation 每次都需要：

  fork agent → 调用 summarization API → 等待响应 → 生成 summary

  Session Memory Compact 只需要：

  直接读 summary.md 文件 → 生成 summary user message → 返回

  没有 API 调用，零延迟、零成本
```

### 9.2 保留消息计算

```typescript
// services/compact/sessionMemoryCompact.ts

calculateMessagesToKeepIndex(messages, lastSummarizedIndex): number

// 从 lastSummarizedMessageId 开始，向后扩展：

// 1. 满足 minTokens（默认 10K）

// 2. 满足 minTextBlockMessages（默认 5 条）

// 3. 不超过 maxTokens（默认 40K）

// 4. floor = 最后压缩边界 + 1（不在压缩边界内切割）
```

**`adjustIndexToPreserveAPIInvariants` 保护**：

API 有两个硬约束，压缩必须满足：

**约束 1：不拆分 tool_use/tool_result 配对**

**约束 2：不拆分共享 message.id 的 streaming 消息**
PS: 这是 streaming 模式下的 API 行为导致的。具体来说，streaming 模式下，一个 assistant 响应是分段到达的。因此，只要保留了一个 message.id 的任意一个块，就必须把该 message.id 的所有块都保留——因为 API 最终会用 message.id 做合并
```
Claude 的 streaming 为每个 content block 发射单独的 assistant 消息：

  Index N:   assistant(id=X), content: [thinking]      ← streaming chunk 1

  Index N+1: assistant(id=X), content: [tool_use: ID]  ← streaming chunk 2

  Index N+2: user, content: [tool_result: ID]

若 startIndex = N+1（只保留 tool_use 但丢弃 thinking）：

  → normalizeMessagesForAPI 按 message.id 合并时

  → thinking 块丢失

  → 模型失去重要的推理过程

正确做法：

  → 检测到 N+1 与 N 共享 message.id

  → 将 startIndex 调整到 N（包含完整的 streaming 组）
```

### 9.3 功能使用判断

触发条件：`effective context window × 90%` (传统压缩触发阈值是effective context window × 93.5%)

多层递进的检查是否可以使用`Session Memory Compact`功能：

```typescript
// 3. 检查环境变量（功能开关）：ENABLE_CLAUDE_CODE_SM_COMPACT（强制启用）

//    或 DISABLE_CLAUDE_CODE_SM_COMPACT（强制禁用）

// 4. 会话记忆文件存在且非空（isSessionMemoryEmpty()）
```

### 9.4 执行流程

```

    ↓
1. 等待任何进行中的会话记忆提取完成（15s 超时）
    ↓

    ↓
```

## 10. 自动压缩（Auto-Compact）

自动压缩是当上下文接近阈值时自动触发的压缩机制，无需用户手动执行 `/compact`。

### 10.1 阈值体系详解

```
┌─────────────────────────────────────────────────────────────────────┐

│                         有效上下文窗口                               │

│                                                                     │

│  0%        warning          autocompact        blocking     100%    │
```

### 10.5 禁用控制矩阵

| `DISABLE_COMPACT=1` | 禁用所有压缩 | 最强 |

| `DISABLE_AUTO_COMPACT=1` | 仅禁用自动压缩，保留 `/compact` | 强 |

| `REACTIVE_COMPACT` feature | 禁用主动压缩，依赖 413 响应 | 中 |

### 10.6 Marble Origami 的特殊性

当 `CONTEXT_COLLAPSE` 开启时，`marble_origami`（ctx-agent）的自动压缩被禁用：

- 若 ctx-agent 上下 blow up，自动压缩触发
- `runPostCompactCleanup` 会调用 `resetContextCollapse()`
- 这会破坏**主线程**的 committed log（共享模块状态）

解决方案：在 `CONTEXT_COLLAPSE` 开启时，ctx-agent 禁用自动压缩。

PS：
- marble_origami 是一个 ctx-agent（context collapse agent），运行在子 agent/fork 中。它和主线程共享同一进程的模块级状态
- marble_origami 是共享进程状态的 ctx-agent，如果它的 auto compact 成功后会触发 resetContextCollapse()——即使加了 isMainThreadCompact 检查，共享状态的存在本身就不安全。所以直接在 shouldAutoCompact 源头把 marble_origami 排除掉，ctx-agent 的上下文爆炸问题由 Context Collapse 系统自己处理，不走 auto compact 路径

## 11. 上下文折叠（Context Collapse）

Context Collapse（`CONTEXT_COLLAPSE` feature flag）是一种**实验性的**上下文管理机制，通过特征门控制。

PS：
- Context Collapse 是一个==已完成实现、正在内部灰度测试的实验性功能，尚未正式发布==。对外既没有文档，也没有稳定的产品化命名
- 由于核心服务源码（services/contextCollapse/）不在 restored-src/ 中，仅通过 14个引用点的交叉分析来推测其完整逻辑

### 11.1 与压缩的核心区别

PS：折叠是用代码实现的，不调用AI
```
压缩（Compaction）：

  丢弃旧消息 → 用摘要替代

  信息损失：中等（通过提示词质量控制）

  延迟：高（需要调用 AI 模型生成摘要）

  精度：低-中等（摘要质量不稳定）

上下文折叠（Context Collapse）：

  选择性丢弃 → 精确控制保留内容

  信息损失：可精确控制

  延迟：低（无需调用 AI）

  精度：高（按比例精确丢弃）
```

### 11.2 设计原理

类似操作系统内存管理的分页策略：
- **90% 阈值**：开始提交（commit）操作——将中间结果写入磁盘
- **95% 阈值**：阻塞新输入——等待提交完成
- **保留策略**：保留最近的 N% 的消息，确保工作上下文不丢失

### 11.3 与自动压缩的互斥

==优先级排序： **Context Collapse > Session Memory Compact >  Auto-Compact**==
```
CONTEXT_COLLAPSE 开启时：
  - Auto-Compact 完全抑制（避免竞争）
  - 90% 提交阈值 < 93.5% 自动压缩阈值
  - 折叠系统在自动压缩之前接管
  - 95% 阻塞阈值 > 93.5% 自动压缩阈值

CONTEXT_COLLAPSE 禁用时：
  - Auto-Compact 正常运作
  - 依赖摘要来管理上下文
```



  - System prompt / appendSystemPrompt
  - Tool result messages（API 返回的原始工具输出） -> ==这个都不被折叠？？那到底折叠些啥==
  - 被 microcompact 压缩后的 cache metadata
  - 最近的 N 条"保护尾"（protected tail）——由 getMessagesAfterCompactBoundary 划定

  折叠的产物：每个被折叠的 span 被替换为一个 metadata placeholder，格式为：

```
<collapsed id="16位collapseId">摘要文本</collapsed>  
```

### 11.6 折叠的两阶段算法

Context Collapse 是**预防性、渐进式**的

 **阶段 1 — Staging（预备折叠）**
  
  触发条件：90% 有效上下文窗口

  有效窗口 tokens ≥ 有效窗口 × 90%

  此时触发：
  1. ctx-agent spawn（子进程，内代号 "marble_origami"）被启动
  2. ctx-agent 分析从"保护尾边界"到当前位置的所有消息 span
  3. **为每个 span 生成摘要**：{ startUuid, endUuid, summary, risk, stagedAt }
  4. 结果写入 staged queue（内存中）
  5. 写入 transcript 一个 marble-origami-snapshot 条目（持久化 staged 队列状态）
  6. staged spans 不会立即从 API 上下文中删除，只是在队列里等待

  ▎ staging 的意义：提前准备好摘要，不等到爆仓才开始处理


  **阶段 2 — Commit（正式折叠）**

  触发条件分两种路径：

  **路径 A（主动）— 95% 阻塞触发**
  有效窗口 tokens ≥ 有效窗口 × 95%

  在下一个 spawn 请求发送前，检查是否已 staged：
  - 如果有 staged → 直接 commit（用已有的摘要，无需再次调用模型）
  - 如果没有 staged → 立即生成摘要后 commit

**路径 B（被动）— 413 溢出恢复**
  API 返回 413 (prompt too long)

  调用 recoverFromOverflow()：
  1. 将 staged queue 中的所有 span 全部 commit（drain）
  2. 用 commit 后的新上下文重试 API 请求
  3. 如果 retry 再次 413 → 降级到 reactiveCompact

## 15. 持久化记忆提取

### 15.1 与会话记忆的区别

## 18. 关键设计决策

### 18.1 为何 fork 子 Agent 执行摘要？

所有后台任务（摘要生成、记忆提取）都通过 `runForkedAgent` 执行，核心原因：

1. **提示缓存复用**：fork 继承主会话的完整缓存前缀（系统提示+工具+上下文消息），避免重复计算
2. **隔离状态**：子 Agent 不能修改主 Agent 的 `readFileState`、skill 列表等内部状态
3. **可中断**：用户 Esc 通过共享的 `AbortController` 信号中断子 Agent
4. **无副作用**：子 Agent 的消息不进入主 transcript（`skipTranscript: true`）

### 18.2 为何微压缩在 API 层而非直接修改消息？

Cached Microcompact 选择在 API 层通过 `cache_edits` 操作，而不直接修改消息内容：

1. **保留提示缓存**：直接修改消息内容会破坏服务器端缓存，导致每次都重新计算
2. **减少网络负载**：API 只需传输增量变更（删除哪些），而非整个前缀
3. **原子性**：服务器端缓存操作是原子的，避免竞态条件
4. **与压缩互补**：微压缩清理旧结果，全量压缩处理累积压力

### 18.3 为何时使用 GrowthBook 动态配置？

所有阈值和开关都通过 GrowthBook 动态配置，而非硬编码：

```typescript
getFeatureValue_CACHED_MAY_BE_STALE('tengu_session_memory', false)

getDynamicConfig_CACHED_MAY_BE_STALE('tengu_sm_compact_config', {})
```

- **零延迟部署**：无需发版即可调整阈值
- **A/B 测试**：不同用户组使用不同配置
- **灰度发布**：逐步扩大功能覆盖
- **缓存非阻塞**：`CACHED_MAY_BE_STALE` 立即返回缓存值，不阻塞初始化

PS：这些参数阈值和开关都是直接暴露出去的接口，可以直接调整而不用变动底层代码
